Assess · Screen 7

Preliminary risk assessment — how each answer is computed

One fixed template, N data bindings. About 8 of 10 questions auto-populate from data the finance center already holds; 2 are qualitative and route to the program office. This screen is the read-only evidence view — where every value came from, its confidence and its rationale. All input — approve, override, qualitative answers, finalize — happens on screen 8.

Assess · Screen 8

Human review & override — the input surface

Auto-populated is not auto-accepted. This is the only place PRA answer decisions happen: a reviewer approves or overrides every answer — overrides capture a reason — the program office enters Q9–Q10, and nothing finalizes without sign-off. (Screen 7 shows how each value was computed.)

Session audit trail

Every decision in this session — rule overrides, exception decisions, approvals, overrides, trigger-config changes, finalizations — logged with kind, date-bearing timestamp (real wall clock for live actions), actor and input refs. Seeded system entries record why each program was flagged at ingestion, against the config in force at ingestion — they are simulated provenance records with a deterministic stamp.
