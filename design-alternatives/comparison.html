Design Alternatives — FEMA Program ID & PRA Automation

Seven working, self-contained re-imaginings of the proof of concept, built on the same data payload and hard constraints but with deliberately different navigation structures, information hierarchies, primary workflows, and interaction models. Each opens directly from disk — no install, no network. Concepts 1–6 are competing product strategies; concept 7 is the post-critique synthesis.

The concepts

Every card links to the working product and its README. Thumbnails are schematic sketches of each concept's information architecture.

Concept 1 · read & decide

The Morning Brief 01-executive-command-center/

An answer-first executive briefing. Plain-English conclusions open the page; every number appears only after the sentence that interprets it; the pipeline survives only as a provenance footnote.

Target user
Acting DCFO / executive sponsor (90 seconds of attention)
Primary workflow
Scan headlines → unfold evidence three levels to transactions → concur / send back / acknowledge → export the briefing with the decision record

Interface: no navigation at all — one scrolling document with in-place depth. Functional: decision capture (with required reasons and undo) becomes the audit trail; tiered affordances — only flagged programs ask for a decision.

Concept 2 · workflow & sign-off

The Assessment Desk 02-risk-assessment-workflow/

The PRA as case work: one case per program per fiscal year, moving through a six-stage lifecycle with ownership, due dates, and hard gates an Inspector General would respect.

Target user
PRA preparer + reviewer/approver
Primary workflow
Trigger creates a case → validate 8 auto answers against evidence → supply Q9/Q10 → route to reviewer → sign off (gated) → export the signed PRA

Interface: queue → case → question; evidence and input are one surface per question. Functional: hard gates — finalize is unreachable until every answer is human-resolved; a chronological decision timeline per case; 3-year comprehensive-cycle due dates.

Concept 3 · interrogate & prove

The Ledger Lens 03-data-exploration-traceability/

One composable analyst instrument: facets, pivots, dual measures side by side, and an always-present lineage inspector — with the inquiry itself recorded as first-class, auditable data.

Target user
Finance-center / program analyst
Primary workflow
Preset or search → slice and compare FY pairs → open code cards, walk the crosswalk → pin evidence → export a watermarked annex with the trail

Interface: search-first, zero pages — every view is a state of one workbench. Functional: append-only replayable investigation trail + pinboard annex export; exceptions and dirty raw codes are always-visible facets.

Concept 4 · ask & adjudicate

The Examiner 04-ai-native-fema/

Agency inverted: the system opens the year with its findings already drafted, declares its computation plan before every answer, volunteers what it could not verify — and the human's job is adjudication.

Target user
Analyst / reviewer who interrogates rather than operates
Primary workflow
Review system-initiated inquiries → ask follow-ups (deterministic question grammar, honestly labeled) → accept / correct / flag every answer → the decision log is the export

Interface: inquiry threads in three panes — no screens, no pipeline. Functional: plan-before-answer, a standing "what I could not verify" ledger, and adjudication controls on every AI output.

Concept 5 · radical alternative

The Chronicle 05-radical-alternative/

Time as the interface: a zoomable chronology where each program is a lane carrying two strands — dollars and transaction volume — and their divergence, not their height, is the anomaly signal.

Target user
Anyone reasoning about change — trajectory, cadence, cause
Primary workflow
Spot the IA lane's strand divergence → zoom to the FY boundary → open the flare's evidence down to transactions → check each lane's 3-year cycle arc → export the chronicle

Interface: pan/zoom on one continuous SVG canvas — fiscal year stops being a dropdown. Functional: the 2024 dual-measure rule as geometry; cycle-due arcs; a viewport-synced ledger as the precise-lookup and screen-reader surface.

Concept 6 · govern the rules

The Crosswalk Registry 06-additional-distinct-concept/

Rules — not spend — as the primary object: every mapping rule with provenance, signatures, a live dollar blast radius, and change control shaped like a code-review system.

Target user
Finance-center data steward + validating SMEs
Primary workflow
Work the validation inbox (inferred rules, exception codes) → confirm with signature → stage rule edits → review the blast-radius diff → second person approves → applied, logged, revertible

Interface: three fixed workspaces under a persistent trust-posture header. Functional: staged diffs showing exactly which transactions re-map and which trigger flags flip before anything applies; % of dollars under confirmed rules — a number no other surface shows.

Concept 7 · integrated hybrid — post-critique synthesis

The Integrity Desk 07-integrated-hybrid/

Answers first, work second, governance underneath — on one audit spine. The brief opens the day; findings deep-link into the case engine and the governed crosswalk; every action lands on one replayable trail.

Target user
OCFO payment-integrity assessment lead (executives are served consumers)
Primary workflow
Read the FY brief → work flagged programs as gated assessment cases → route data problems through staged, second-person-approved changes → export briefing, signed PRAs, change record

Combined: 01's briefing register + 06's trust posture in the situation bar + 02's case gates + 04's plan-before-answer evidence format + 05's dual-strand lane idiom + 03's trail semantics. Left out on purpose: the full canvas, the ask-anything grammar, the nine-facet bar.
