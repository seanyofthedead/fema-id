## 3. The load-bearing distinction: public *funding* data vs. actual *spend* data

The transcript is explicit (~14:47): public information exists **"as far as funding goes, but … the actual spend is what we're looking at,"** and because disaster money is typically **no-year money**, "the actual amount to get spent in a year is going to be hard to reconcile to the budget."

Live verification confirms this boundary precisely:

| What the public can see (verified) | What the client's problem needs (not public) |
|---|---|
| **Obligations** at project grain — `federalShareObligated`, `totalObligated`, `projectAmount` per project worksheet (SRC-03) or subgrant (SRC-04), tied to real disaster numbers (SRC-02). | **Disbursements/actual spend by internal program ID** per fiscal year — the measure that feeds the PRA and the 20% YoY trigger (REQ-006/010; ASSUMP-05). |
| **Awards and account balances** government-wide (SRC-05), budget authority (SRC-08), Treasury-level cash/outlays (SRC-11). | The **code→sub-program→program rollup** and the adjustment rules applied "behind the scenes" (REQ-001/003/004). |
| **Annual program-level improper-payment aggregates** (SRC-06). | The **10-question PRA instrument** and its per-question data bindings (REQ-007; ASSUMP-04). |

Every public dollar figure available to us is on the *funding/obligation* side. **No public source provides FEMA's internal disbursement-by-program-ID ledger, and none of it may be fabricated as if real.** That forces — and shapes — the synthetic-data strategy below.

## 4. Recommended synthetic-data strategy (grounded only in verified sources)

1. **Real skeleton, synthetic flesh.** Use live-verified public identifiers as the immutable skeleton: disaster numbers and dates from SRC-02 (DR-4332 Harvey/TX, DR-4337 Irma/FL, DR-4339/4340 Maria/PR/VI, DR-4346 Irma/SC), program families and assistance-listing numbers from SRC-03/SRC-12 (Public Assistance 97.036) and SRC-04 (HMGP/BRIC/FMA program areas). Everything at finer grain than public data is synthetic and watermarked as such.
2. **Calibrate magnitudes to verified obligations.** Pull per-disaster, per-program obligation distributions from SRC-03/SRC-04 (both refresh daily; spot-checked live for DR-4332). Generate a synthetic *disbursement* ledger whose totals draw down against those obligation envelopes over multiple fiscal years — modeling the no-year-money behavior the transcript describes — so demo numbers are plausible in scale without claiming to be real spend (ASSUMP-05/ASSUMP-10).
3. **Invent the code anatomy openly.** Synthetic internal codes (fund/program/event segments) are declared fictional in the demo. The event segment references real disaster numbers (SRC-02) so the Harvey/Irma/Maria grouping story (REQ-005) lands with a recognizable anchor. The undocumented mapping rules (REQ-001) are represented by a *configurable* rule set the demo "discovers" via the historical-mining approach (REQ-013) across ≥4 synthetic fiscal years (ASSUMP-07).
4. **Bind the PRA to the regime, not to FEMA's form.** The synthetic 10-question PRA (ASSUMP-04) is labeled illustrative and structured around the PIIA/M-21-19 risk-assessment concepts (SRC-07), with ~8 quantitative bindings computed from the synthetic ledger (incl. the configurable 20% YoY trigger, ASSUMP-03) and 2 qualitative stubs (REQ-009).
5. **Narrative garnish from oversight sources only as verified quotes.** PaymentAccuracy.gov rates (SRC-06), GAO high-risk framing (SRC-10), and OIG PIIA findings (SRC-09) may appear on slides **only after the specific figure/quote is pulled from the primary document** — each is currently verified at title/URL level, with contents marked *unverified — validate*.
6. **Hard rule carried from the constraints:** no fabricated FEMA internals — no invented SOP text, no "real" code mappings, no actual-spend figures presented as genuine. Synthetic artifacts carry a visible watermark.
