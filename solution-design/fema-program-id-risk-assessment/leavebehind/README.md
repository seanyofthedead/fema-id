# Leave-behind: single-file HTML demo

**Package:** FEMA Program ID & PRA Automation (demo)
**Document date:** 2026-07-08
**Status:** Conceptual demo on **synthetic, watermarked data** (`SYNTHETIC-DEMO`). Not production; no real FEMA spend, no PII.

`fema-demo.html` is the portable variant of the 10-screen demo storyboard (file 11): **one self-contained HTML file** that can be emailed and double-clicked on a locked-down machine. No server, no install, no build step at demo time, no network.

---

## 1. How to open

1. Copy `fema-demo.html` anywhere (email attachment, USB, shared drive).
2. Double-click it — it opens in any modern browser (Edge, Chrome, Firefox, Safari).
3. That's it. It renders identically with wifi fully off.

**Reset to a clean state = reload the page** (F5; once you have made session decisions the browser asks for confirmation first, so sign-off work is never lost to an accidental refresh). All state is in browser memory only — the file uses no `localStorage`/`sessionStorage`, writes nothing to disk, and makes zero network requests (any URL you see on screen is a provenance citation, inert text).

### 2a. Mapping-visualization features (added 2026-07-09)

Four features from `../enhancements/MAPPING_DEMO_IDEAS.md` §4 make the ID→program linking itself visible — all deterministic re-use of the existing in-file engine, **no new generator fields, no answer-key use**:

- **Follow the dollar (FLOW-01):** click any transaction (screen 2 *Trace →* buttons, screen 3 seeded chips, code-search card, or a flow-map code node) and watch it walk raw code → cleansing rule → sub-program rule → rollup → event tag → the totals and PRA lines it feeds, each hop citing its rule ID, status and confidence. Exception codes stop at the queue with the labeled AI suggestion.
- **Flow map (FLOW-02):** screen 3's second tab — dollar-weighted ribbons across four fixed columns (codes / sub-programs / program / disaster events), hover for exact figures, click a code to trace; the lineage overlay's *Open in flow map* button lights the traced path.
- **Code search (FLOW-04):** top-bar search that resolves any spelling — canonical, format-dirty (`pa/97036/4332`), or retired legacy alias (`LEG-0001`) — through the embedded cleansing rules to a full card: code anatomy, mapping chain with rule + confidence, per-FY spend bars, DR chips.
- **What-changed drawer (FLOW-07):** every rule edit or exception approval opens a drawer enumerating exactly what moved (before/after snapshot diff of the deterministic engine), with Keep (audited) or an exact Revert.
- **Crawler holdout reveal (FLOW-05, added 2026-07-09):** screen 3's third tab (*Inference test*). The historical-mining crawler reads **only the FY2022–2025 mapping history** (396 assignments; the split is by fiscal year, constant `F5_HOLDOUT_FY` in the template) plus the unlabeled FY2026 intake transactions, proposes FY2026 groupings blind, then a **Reveal** scores the proposals against the held-out FY2026 mapping rows already embedded as ordinary data. Confidence is **evidence-weighted** (cross-year agreement × observation depth × raw-identity stability, capped 0.97), so it varies per code and drops exactly where the raw extract churned: 95 codes clear the 0.85 routing bar; 4 codes — three retired legacy aliases plus one sparse dirty code — fall below it and are *caught by review*; the 6 never-seen codes get only a low-confidence structural-similarity guess (event mix, fund, segment nearness, amount profile) and route to the exception queue. Inference (`f5Infer`) and scoring (`f5Score`) are separate code paths — the inference function never receives the holdout labels. The accuracy headline is computed at reveal time, never hardcoded. **No new generator fields; the validation-only answer-key file remains unread and un-embedded (DEC-22)** — the runtime holdout truth is the planted FY2026 mapping that was always in the file. Note the honesty caveat in `DEMO_SCRIPT.md`: this dataset's grouping history is deliberately stable, so the zero-miss auto-classification result is expected by construction.

## 3. How "AI" works in this file (important)

A single offline file cannot safely make live LLM calls (no network; keys can't be embedded). So, per the package's own design rule — *deterministic code owns every reportable number; AI only proposes, explains, scores* (file 09) — the demo:

- computes **every reportable number** (rollups, YoY deltas, trigger flags, PRA quantitative answers) in deterministic embedded JavaScript over the embedded ledger;
- ships **precomputed, deterministic plain-language explanations** and confidence scores where the product would use an LLM. Every such element is labeled **"AI-SUGGESTED · PRECOMPUTED FOR DEMO"** and is human-overridable in-session;
- grounds rationale text in public guidance citations only (OMB M-21-19 / A-123 App. C, PaymentAccuracy.gov — ASSUMP-18); nothing internal is fabricated.

In the connected product (file 09), those same slots are filled by live historical mining, similarity matching, LLM rationale and RAG — behind the same guardrails.

## 4. Data disclaimers (say these out loud when demoing)

- Every record is **synthetic** and watermarked `SYNTHETIC-DEMO` — verified at build time for every embedded row.
- Dollar figures are synthetic **disbursements** calibrated to public FEMA **obligation** envelopes (SRC-03/SRC-04). Obligations are funding, not spend; the demo never equates them (ASSUMP-05, ASSUMP-10).
- The DR numbers (4332 Harvey, 4337/4338/4341/4346 Irma, 4339/4340 Maria) are **real public identifiers** (SRC-02); all dollars against them are synthetic.
- The 10-question PRA is an **illustrative placeholder** (ASSUMP-04); FEMA's real instrument is pending (SME-05).
- The 20% trigger is **configurable, on-screen**, because the transcript hedged it (REQ-010, SME-01).

## 5. Regenerating the file

`fema-demo.html` is generated — don't edit it by hand. When the Wave 1 dataset changes:

```
cd solution-design/fema-program-id-risk-assessment/leavebehind
python build_demo_html.py        # stdlib only, Python 3.9+
```

The script reads `../data/synthetic/*.csv` + `../data/generator/rules.yaml`, verifies the watermark on every row, inlines the data as JSON into `template.html`, and writes `fema-demo.html`. Output is **deterministic** (no timestamps, no randomness): re-running on unchanged data is byte-identical. The build refuses to run if the template contains any runtime-fetched URL, `fetch`/XHR, or browser-storage API — and it asserts the validation-only answer key is neither read nor referenced (DEC-22).

Files here:

| File | Role |
|---|---|
| `fema-demo.html` | **The deliverable** — generated, self-contained |
| `template.html` | App shell (HTML/CSS/JS) with a data placeholder; source of truth for the UI |
| `build_demo_html.py` | Deterministic builder + guardrail checks |
| `DEMO_SCRIPT.md` | 5–8 minute presenter click path |

## 6. Known scope notes (stubbed vs real)

- **Real (computed live in the file):** cleansing counts, all rollups, event splits, YoY math, trigger flags, PRA Q1–Q8 bindings, exception-queue effects on totals, exports, parity check.
- **Precomputed/deterministic stand-ins:** AI rationales, similarity suggestions and confidence scores (labeled as such); the connected build replaces them with live inference (file 09).
- **Illustrative:** the PRA instrument (ASSUMP-04), program taxonomy beyond PA/HM (ASSUMP-09), roles (ASSUMP-19).
- **Out of scope for the leave-behind:** XLSX/PDF binary export (CSV/HTML/JSON are generated client-side instead — Excel opens the CSV directly; print the HTML report to PDF); live LLM/RAG calls; persistence.
